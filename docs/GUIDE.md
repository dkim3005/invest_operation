# PoolDesk 가이드 — 쉽게 이해하기

이 문서는 PoolDesk를 **용어를 몰라도 이해할 수 있게** 풀어 쓴 가이드입니다.
용어 사전 + 어떤 데이터를 모으는지 + 어떻게 쿼리하는지 + 무슨 로직이 도는지를
한곳에 모았습니다.

---

## 0. 한 줄 요약

> 자산운용사가 **매일 손으로 하던 백오피스 업무**(데이터 점검 → 장부 대조 →
> 자산가치 계산 → 문제 처리 → 보고서)를 **명령 한 번으로 자동화**하는 프로그램.

---

## 1. 큰 그림 — 무슨 문제를 푸나

IMCO 같은 자산운용사는 여러 고객(연금·보험 기금 등)의 돈을 **한데 모아** 굴립니다.
주식만 모은 바구니, 채권만 모은 바구니… 이렇게 자산 종류별로 나눠 담죠.

이 돈이 굴러가려면 **운영팀(Investment Operations)**이 매 영업일 이런 일을 합니다:

1. 외부에서 데이터를 받는다 (가격, 우리 보유 내역, 거래 내역…)
2. 그 데이터가 멀쩡한지 점검한다
3. **우리 장부**와 **자산을 보관하는 은행의 장부**가 일치하는지 대조한다
4. 바구니마다 지금 얼마짜리인지(NAV) 계산한다
5. 안 맞는 항목을 조사해서 담당팀에 넘긴다
6. 경영진·고객에게 보고서를 낸다

→ 반복적이고 느리고 실수 나기 쉬운 일. **PoolDesk가 이걸 통째로 자동화**합니다.

---

## 2. 용어 사전

면접에서 들을 영어 용어 → 쉬운 뜻. **굵은 영어**가 실제 쓰는 말입니다.

### 구조

- **Asset manager (자산운용사)** — 남의 돈을 대신 굴려주는 회사. 예: IMCO.
- **Investment Operations (투자운영)** — 거래가 *체결된 뒤*의 모든 뒷정리를
  맡는 팀. 흔히 "백오피스/미들오피스"라 부름. 거래를 *하는* 게 아니라, 거래
  결과를 정확히 기록·검증·보고하는 일.
- **Pool (풀)** — 여러 고객의 돈을 자산 종류별로 모은 바구니. PoolDesk엔 5개:
  주식·채권·사모·부동산·인프라 풀.
- **Client (클라이언트)** — 그 바구니에 돈을 넣은 고객(연금기금 등).
- **Unit (유닛)** — 바구니의 "지분 한 조각". 고객은 유닛을 보유함.
- **Unit price (유닛가격)** — 지분 한 조각의 값. `풀 전체 가치 ÷ 발행 유닛 수`.
- **Transfer agency (이전대리)** — 누가 유닛을 몇 개 갖고 있는지 적어두는 장부.
  PoolDesk에선 `client_allocation` 표가 그 역할.

### 가치 평가

- **NAV (Net Asset Value, 순자산가치)** — 바구니의 **지금 총가치**.
  `가진 자산의 시장가치 + 현금 − 부채`. 펀드 세계에서 제일 중요한 숫자.
- **Position (포지션)** — "무엇을 몇 개 갖고 있다". 예: A주식 1,000주.
- **Cost basis (장부원가)** — 그 자산을 살 때 낸 값. 시장가와 비교용.
- **Mark-to-market (시가평가)** — 보유 자산을 *현재 시장가*로 다시 계산하는 것.
- **Management fee (운용보수)** — 운용사가 떼는 수수료. 매일 조금씩 쌓임(accrual).

### 거래·보관

- **Trade (거래)** — 사고팔기. `BUY`/`SELL`.
- **Settlement (결제)** — 거래 후 *실제로* 돈과 증권이 오가는 것. 보통 거래 +2일
  (T+2). 아직 안 끝났으면 `PENDING`, 실패하면 `FAILED`.
- **Custodian (커스터디언)** — 자산을 실제로 보관하는 은행. 운용사에 매일
  "당신은 이걸 이만큼 갖고 있다"는 장부를 보냄.

### 점검·대조

- **Data quality (데이터 품질)** — 받은 데이터가 멀쩡한지. 빠졌나? 오래됐나?
  말도 안 되는 값인가?
- **Stale price (스테일 가격)** — 갱신 안 된 **오래된 가격**. 값 자체는 정상이라
  눈에 안 띄지만 위험.
- **Outlier (이상치)** — 말이 안 되는 값. 0 이하거나 하루에 ±25% 넘게 튄 가격.
- **Reconciliation (대사)** — **두 장부를 한 줄씩 맞춰보는 것.** PoolDesk의 핵심.
  우리 장부 vs 커스터디언 장부.
- **Break / Exception (불일치 / 예외)** — 대사에서 발견된 **안 맞는 항목**.
  - `QUANTITY_MISMATCH` — 양쪽 다 있는데 수량이 다름
  - `MISSING_IN_CUSTODIAN` — 우리 장부엔 있는데 커스터디언엔 없음
  - `MISSING_IN_INTERNAL` — 커스터디언엔 있는데 우리 장부엔 없음
- **Severity (심각도)** — break가 **금액으로 얼마나 큰지**. HIGH(≥100만 CAD) /
  MEDIUM(≥10만) / LOW. 건수가 아니라 금액으로 우선순위를 매김.
- **Maker-checker (메이커-체커)** — 한 명이 입력, 다른 한 명이 검증하는 통제 원칙.

### 기타

- **FX (환율)** — 외화를 자국 통화(여기선 CAD)로 바꾸는 비율. 미국 종목은
  USD라서 CAD로 환산해야 함.
- **P&L (Profit & Loss, 손익)** — 하루 사이 NAV가 얼마 늘었나/줄었나.
- **Feed (피드)** — 외부 시스템에서 매일 자동으로 들어오는 데이터 흐름.
- **AI 예외 분류** — LLM(Claude)이 각 break를 보고 *추정 원인·담당팀·우선순위·
  해소 메모*를 제안하는 것. **사람 대신 결정하지 않고, 검토를 도와줄 뿐.**

## 3. 어떤 자료를 수집하나

PoolDesk가 다루는 데이터는 두 종류입니다.

### 참조 데이터 (거의 안 변함) — `data/reference/`

| 파일 | 내용 (쉽게) |
|---|---|
| `securities_master.csv` | 다루는 종목 34개 목록 (이름·통화·소속 풀) |
| `pools.csv` | 5개 풀 정보 (이름·자산군·운용보수율) |
| `clients.csv` | 고객 5곳 (가상) |
| `client_allocations.csv` | 어느 고객이 어느 풀에 유닛 몇 개 (= transfer agency 장부) |

### 일별 피드 (매 영업일 새로 들어옴) — `data/feeds/{날짜}/`

| 파일 | 내용 (쉽게) |
|---|---|
| `market_prices.csv` | 그날 종목별 가격 |
| `fx_rates.csv` | 그날 환율 (USD→CAD) |
| `internal_positions.csv` | **우리 장부** — 무엇을 몇 개 보유 |
| `custodian_positions.csv` | **커스터디언 장부** — 보관 은행이 본 보유 내역 |
| `trades.csv` | 그날 거래 내역 |
| `cash_ledger.csv` | 풀별 현금 입출금 |

> **솔직한 점**: 가격·환율은 실제로 받을 수 있지만(yfinance), 운용 장부·거래·
> 커스터디언 기록은 **기밀이라 공개 데이터가 없어서 합성**합니다. 일부러 결함도
> 심어 넣습니다(빠진 가격, 튄 가격, 안 맞는 수량…) — 점검·대사 엔진이 잡을
> 거리를 주려고요.

## 4. 데이터를 어떻게 저장하고 쿼리하나

- **저장**: 위 CSV들을 **SQLite 데이터베이스**(`pooldesk.db`) 하나에 적재합니다.
  표는 두 종류 — `dim_*`(이름·정보 같은 차원), `fact_*`(가격·포지션 같은 수치).
- **쿼리**: 평범한 **SQL**로 묻습니다. 핵심 질문들은 `sql/checks.sql`에 모아뒀습니다.

쉬운 예시 (실제 쿼리를 말로 풀면):

| 알고 싶은 것 | SQL이 하는 일 (말로) |
|---|---|
| 오늘 가격이 빠진 종목? | 보유 종목 목록과 가격 목록을 맞춰보고, 가격 쪽에 없는 걸 골라냄 (`LEFT JOIN … IS NULL`) |
| 우리 장부 vs 커스터디언 차이? | 두 표를 (풀, 종목) 기준으로 붙여서 수량이 다르거나 한쪽에만 있는 행을 찾음 |
| 중복으로 들어온 가격? | (종목, 날짜)로 묶어서 2건 이상인 것 (`GROUP BY … HAVING COUNT(*)>1`) |
| 풀별 자산가치? | 수량 × 가격 × 환율을 풀별로 합산 (`SUM … GROUP BY pool`) |

> Python 엔진과 똑같은 로직을 SQL로도 써둔 이유 — *"도구가 아니라 논리가
> 핵심"*임을 보여주려고. 면접에서 SQL 질문이 나오면 이 파일이 답안지입니다.

## 5. 매일 무슨 계산이 도나 — 6단계 로직

명령 `python main.py run-all`을 치면, 하루치로 이 6단계가 순서대로 돕니다.

1. **수집 (Ingest)** — 그날 6개 피드 파일을 DB에 적재. 한 파일이라도 깨졌으면
   그날 전체를 안 들임 (반쪽짜리 적재 방지).
2. **데이터 품질 (Data Quality)** — 7가지 검사를 돌려 0~100점 매김. (스키마·결측·
   스테일·이상치·중복·참조무결성·환율 누락)
3. **대사 (Reconciliation)** — 우리 장부 vs 커스터디언을 (풀, 종목)으로 맞춰봄.
   안 맞으면 break로 기록하고, **금액 영향**으로 심각도를 매김.
4. **NAV 계산** — 풀별로 `보유자산 시장가치 + 현금 − 운용보수 = NAV`. 유닛가격과
   고객별 보유가치, 하루 손익도 계산. (가격이 이상하면 직전 정상가로 대체 —
   나쁜 값이 NAV를 오염시키지 않게)
5. **AI 예외 분류** — 열린 break 하나하나를 LLM이 보고 *추정 원인·담당팀·
   우선순위·해소 메모*를 채움. 키가 없으면 규칙 기반으로 대체. **break는 계속
   "열림(OPEN)" 상태 — 사람이 닫음.**
6. **리포팅** — 일일 Excel 운영팩(7탭)·PDF 요약·HTML 대시보드·Power BI용 CSV
   생성.

각 단계는 **다시 돌려도 안전(idempotent)** — 같은 날을 재실행하면 그날 데이터를
지우고 다시 넣으므로 중복이 안 생깁니다.

## 6. 결과물은 어디서 보나

`python main.py run-all` 후 `reports/` 폴더에:

- `dashboard.html` — **브라우저로 여는 대시보드** (제일 보기 쉬움)
- `daily/PoolDesk_Ops_*.xlsx` — 일일 운영팩, 7개 탭
- `daily/PoolDesk_Summary_*.pdf` — 한 장 요약
- `weekly/`, `monthly/` — 주간·월간 롤업
- `powerbi/*.csv` — Power BI 연결용 데이터
- `pipeline.log` — 각 단계 실행 기록

## 7. 면접에서 한 문장으로

> *"I built a working simulation of an Investment Operations daily workflow —
> it ingests feeds, runs data-quality checks, reconciles our book against the
> custodian, computes NAV, uses an LLM to triage the exceptions, and produces
> the daily reports — all automated, so the team only reviews what's flagged."*

한국어로: *"투자운영팀이 매일 손으로 하는 일 — 데이터 점검, 장부 대사, NAV
계산, 예외 처리, 리포팅 — 을 하나의 자동 파이프라인으로 만들었고, AI가 예외를
1차 분류해 사람은 표시된 것만 보면 되게 했습니다."*
